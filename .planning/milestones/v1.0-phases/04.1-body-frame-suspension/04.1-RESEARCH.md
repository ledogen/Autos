## Summary

Phase 4.1 is a surgical coordinate-frame correction to the Phase 4 suspension. No new physics concepts are introduced; only the reference frame in which existing spring/damper forces act is corrected. The hub remains a 1D rigid slider — the only degree of freedom is displacement along the body's local strut axis (body-Y). The three architectural changes are: (1) replacing the world-Y hub ODE with a strut-axis hub ODE, (2) routing the full 3D contact normal — currently only the world-Y component reaches the body — by splitting it into a strut-axis component (existing spring pathway) and an X/Z residual (new direct body force), and (3) fixing the wheel mesh visual binding to use world-space hub position inverse-transformed into body-local space rather than approximating world-Y delta as body-local delta.

The contact normal routing fix (D-06) is the highest-risk change because it touches the force-accumulation contract between `suspension.js` and `physics.js`. The current guard at `physics.js:270` exists specifically to prevent double-counting the contact normal on flat ground. Phase 4.1 replaces this guard with a structurally clean split: the strut-axis component feeds the spring (body sees it indirectly through `_suspForceAccum`), and the X/Z residual feeds a new `_hubNormalXZ` accumulator that physics.js applies directly. On flat ground the X/Z residual is exactly zero, so flat-ground behavior is preserved and all existing test assertions remain valid.

All four existing assertion scenarios (m4-02, m4-04, m4-05, m4-06) run on flat ground with an upright body, so `dot(contact_normal, body_up) = 1.0` and `_tireFz` equals the old `tireFz`. The assertions check `fl_fz + fr_fz + rl_fz + rr_fz` against `m*g` — this sum is unchanged on flat ground. The new m4-07 ramp-slide scenario specifically exercises the X/Z routing fix.

**Primary recommendation:** Work from the inside out — suspension.js strut-axis ODE first, then physics.js accumulator application, then visual binding, then params/debug/logger plumbing. Each layer can be tested incrementally against flat-ground scenarios before the ramp scenario is needed.

## Architecture Patterns

### Accumulator Contract (inherited, extended)

The existing pattern: `suspension.js` zeroes accumulators at the start of each outer step, accumulates across N substeps (`+= value / N`), physics.js consumes them after the substep call.

Phase 4.1 adds one new accumulator:
- `params._hubNormalXZ[i]` — a `THREE.Vector3` (or `{x,y,z}` plain object if staying pure-math) per corner, carrying the X/Z residual contact normal force. Zero when flat. Applied in `physics.js` after `stepSuspensionSubsteps` returns.

**Initialization pattern** (must match existing):
```javascript
// In main.js (alongside the existing transient scratch arrays):
RANGER_PARAMS._hubNormalXZ = [
  { x: 0, y: 0, z: 0 },
  { x: 0, y: 0, z: 0 },
  { x: 0, y: 0, z: 0 },
  { x: 0, y: 0, z: 0 }
]
```

**Zero pattern in suspension.js** (must match existing zero pattern):
```javascript
for (let i = 0; i < 4; i++) {
  params._tireFz[i] = 0
  params._suspForceAccum[i] = 0
  params._hubNormalXZ[i].x = 0
  params._hubNormalXZ[i].y = 0
  params._hubNormalXZ[i].z = 0
}
```

### Pure-Math Contract (`suspension.js`)

`suspension.js` MUST NOT import Three.js. The injected helper pattern (`params._rotateVector`) must be extended:

- `params._rotateVector(v)` — already exists, rotates body-local → world
- Phase 4.1 adds `params._inverseRotateVector(v)` — not needed inside suspension.js (hub ODE only needs strut-axis scalars, and body_down is computed via `_rotateVector({x:0,y:-1,z:0})`). The wheel mesh inverse-transform lives in `main.js` where Three.js is available directly.

**Critical:** `suspension.js` can derive `body_down_world` from the existing helper:
```javascript
const body_down = params._rotateVector({ x: 0, y: -1, z: 0 })
// body_down is a {x,y,z} plain object (the return type of _rotateVector)
```

### Hub World Position (derived, not state)

```javascript
// D-02: derived hub world position for each corner
// strutComp[i] positive = compressed (hub closer to body than rest)
// (L_S - strutComp) = current strut length
const strutLen = L_S - vehicleState.strutComp[i]
const hubWorldX = mountWorldX + strutLen * body_down.x
const hubWorldY = mountWorldY + strutLen * body_down.y
```

On flat ground (body upright): `body_down = (0,-1,0)`, so:
`hubWorldY = mountWorldY - strutLen` — which is exactly the old formula `mountWorldY - (L_S - strutComp)`.

### Suspension Travel Stop Force Pattern

```javascript
// D-08 / D-09: bump and droop stops — linear penalty springs
// Inside the per-corner force loop in stepSuspensionSubsteps:
const travel    = isFront ? params.suspensionTravelFront : params.suspensionTravelRear
const DROOP     = params.DROOP_STOP_STIFFNESS || 20000  // constant

// Bump stop: engages when compressed past travel limit
const bumpOvershoot  = strutComp - travel
const bumpStopForce  = bumpOvershoot > 0 ? -params.bumpStopStiffness * bumpOvershoot : 0
// Negative because it pushes hub away from body (reduces compression)

// Droop stop: engages when extended past zero strutComp (droopRange = amount below rest)
// Simplified: stop at strutComp <= 0 (fully extended)
const droopOvershoot = -strutComp  // positive when extended past zero
const droopStopForce = droopOvershoot > 0 ? DROOP * droopOvershoot : 0
// Positive because it pulls hub back toward body (increases compression)

// Sum into F_total:
const F_total = Fn_strut - springTerm - c_S * strutCompVel + arbF[i]
              - m_u * 9.81 * gravDot    // D-05: gravity component along strut
              + bumpStopForce + droopStopForce
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Quaternion inverse | Manual conjugate math | `THREE.Quaternion.clone().invert()` | Three.js normalizes correctly; manual conjugate fails if quat drifted |
| Cross product | `ax*by-ay*bx` etc. | `THREE.Vector3.crossVectors(a, b)` | Three.js handles edge cases (zero vectors) |
| 3D dot product | Manual `a.x*b.x+...` | `THREE.Vector3.dot(b)` | Cleaner, same performance |
| Vector projection onto axis | Manual decomposition | `v.dot(axis) * axis` (for component along axis) | Standard decomposition; `axis` must be normalized |

**Key insight:** The body_up and body_down vectors used throughout are already unit vectors (quaternion rotation of unit vectors is unit). No normalization needed after `applyQuaternion`.

## Common Pitfalls

### Pitfall 1: Forgetting the suspForceAccum Direction Change
**What goes wrong:** `physics.js:176` currently applies `_suspForceAccum[i]` as a world-Y force: `new THREE.Vector3(0, params._suspForceAccum[i], 0)`. After Phase 4.1, the spring acts along the strut axis (body_up_world direction). If this line is not updated, the spring force still acts in world Y on a pitched body — the exact bug being fixed.
**Why it happens:** The accumulator variable name is unchanged; the bug is invisible in the force-accumulation code itself.
**How to avoid:** Change to `body_up.clone().multiplyScalar(params._suspForceAccum[i])` where `body_up` is the pre-computed `new THREE.Vector3(0,1,0).applyQuaternion(...)` from Step 2. The torque arm `rMount × suspBodyForce` calculation follows automatically since `suspBodyForce` is now a 3D vector.
**Warning signs:** Car behaves correctly on flat ground but "sinks" sideways when pitched — strut spring not pushing along strut.

### Pitfall 2: D-06 Double-Count on Flat Ground
**What goes wrong:** If `_hubNormalXZ[i]` is accidentally accumulated as a world-Y force (not X/Z only), the body receives the contact normal twice on flat ground (via spring and via direct force).
**Why it happens:** The X/Z residual on flat ground is exactly zero, so this bug would be invisible on flat ground and only manifest on slopes.
**How to avoid:** Verify on flat ground that `|_hubNormalXZ[i]|` is zero (or near machine epsilon). Add an assertion in the stability-check block during development.
**Warning signs:** Car bounces violently when placed on a ramp but is fine on flat ground.

### Pitfall 3: strutComp Static Equilibrium Formula
**What goes wrong:** `computeStaticEquilibrium` currently uses `suspComp = (cornerMass - wheelMass) * g / k_S` for the suspension spring. D-11 specifies `strutComp[i] = m_sprung_corner * g / k_S_i` where sprung mass = `mass * weight_i / 2` (excludes wheelMass). These formulas give different values.
**Why it happens:** The Phase 4 formula included the sprung mass only. D-11 matches this but uses different nomenclature. Verified numerically: both produce `strutComp ≈ 111 mm` at current params.
**How to avoid:** Use `(mass * (isFront ? weightFront : weightRear) / 2) * g / k_S` as D-11 specifies. Verify against 111 mm at current params.
**Warning signs:** Car spawns visibly too high or too low; `strutComp` at frame 0 is far from equilibrium.

### Pitfall 4: strutCompVel Sign Convention
**What goes wrong:** In Phase 4, `suspVel = hubVy[i] - mountVelY` (positive = compression, hub moving toward mount). D-03 specifies `strutCompVel = mountVelStrut - hubVelAlongStrut` (also positive = compressing). The sign convention must be internally consistent: the spring term `k_S * strutComp` must push hub away from body (negative contribution to hub acceleration), and damper `c_S * strutCompVel` must resist compression.
**Why it happens:** Sign errors in the damper term are the most common suspension physics bug (Phase 4 had one — `hubVy - mountVelY` was inverted from `mountVelY - hubVy`).
**How to avoid:** Derive from first principles: `F_spring_on_hub = -k_S * strutComp` (pushes hub away from body when compressed). `F_damper_on_hub = -c_S * strutCompVel` (resists compression when strutCompVel > 0). Hub ODE: `m_u * strutCompVelDot = Fn_strut - k_S*strutComp - c_S*strutCompVel + arbF - m_u*g*gravDot + stops`. Body sees the Newton-3rd pair: `F_body = k_S*strutComp + c_S*strutCompVel`.
**Warning signs:** Suspension oscillates unstably from rest; springs appear to attract rather than repel.

### Pitfall 5: Gravity-on-Hub Tapering
**What goes wrong:** D-05 says gravity on hub along strut = `m_u * g * dot(world_up, body_up)`. When body is upright this is `m_u * g` (full gravity). When body is horizontal it is zero. If this term is omitted and only `m_u * g` (world Y) is used, the hub dynamics are wrong on pitched bodies.
**Why it happens:** The old code had `-m_u * 9.81` (constant world gravity), which worked because world-Y was the strut axis. After the frame change, the effective gravity along the strut depends on body orientation.
**How to avoid:** `const gravDot = body_up.x * 0 + body_up.y * 1 + body_up.z * 0` = `body_up.y` (Y component of body_up). This is already `dot((0,1,0), body_up)`. Use `gravDot` in the hub ODE: `-m_u * 9.81 * gravDot`.
**Warning signs:** On a perfectly horizontal body (body rolled 90°), hub still accelerates vertically instead of staying stationary relative to body.

### Pitfall 6: _suspStabChecked Not Reset After bumpStop Change
**What goes wrong:** The stability check runs once and sets `params._suspStabChecked = true`, never re-running even if slider values change. If `bumpStopStiffness` is raised via slider to extreme values, no warning fires.
**Why it happens:** The flag was designed for startup verification, not live slider changes.
**How to avoid:** The Phase 4.1 design-time verification (confirmed in research) shows that even at `k_bump = 10 * k_main` (363,000 N/m effective), the stability criterion `sdt^2 * k/m = 0.35 << 4` holds safely. The existing check does not need to include `k_bump` because the effective frequency is still well within the sdt budget. Document this in a code comment rather than expanding the check.
**Warning signs:** Hub oscillates violently only when bump stop is engaged. Resolution: raise N.

### Pitfall 7: Wheel Mesh Still Uses Old Position After Visual Fix
**What goes wrong:** `syncMeshesToState` currently has TWO code paths for wheel position: the `wheelLocalOffsets` rest position and the hub-Y deviation override (line 257-259). After D-07 fix, the deviation override must be replaced entirely — not just the Y component. X and Z must also come from the derived hub world position (especially important when `suspensionBodyOffsetFront/Rear` is nonzero — the mount shifts in body space).
**Why it happens:** The X/Z of the wheel is currently hardcoded to `wheelLocalOffsets[i].x/z`. Once `suspensionBodyOffsetFront/Rear` is added, the mount Y shifts, which changes where the hub ends up in world space.
**How to avoid:** Compute full hub world position via D-02 formula, then inverse-transform entire `{x,y,z}` into body-local space. Set `wheelMeshes[i].position.copy(hubLocal)`.
**Warning signs:** Wheels at correct Y but wrong X or Z when body-offset slider is non-zero.

### strut_comp_vel — mount velocity projection
```javascript
// [VERIFIED: derived from existing mountVelY formula in suspension.js:258-259]
// Full mount velocity (3D):
const mountVelX = vehicleState.velocity.x +
  (vehicleState.angularVelocity.y * rMount.z - vehicleState.angularVelocity.z * rMount.y)
const mountVelY = vehicleState.velocity.y +
  (vehicleState.angularVelocity.z * rMount.x - vehicleState.angularVelocity.x * rMount.z)
const mountVelZ = vehicleState.velocity.z +
  (vehicleState.angularVelocity.x * rMount.y - vehicleState.angularVelocity.y * rMount.x)
// Project onto strut axis (body_down):
const mountVelStrut = mountVelX * body_down.x + mountVelY * body_down.y + mountVelZ * body_down.z
// Hub velocity along strut:
// hubVelStrut = d/dt(strutComp) = strutCompVel (positive = compressing)
// Wait: strutComp = L_S - (mount_pos - hub_pos).dot(body_down)
// d(strutComp)/dt = -d/dt[(mount-hub).dot(body_down)]
// = -(mountVelStrut - hubVelStrut) + correction for body_down rotation
// Small-mass approximation: ignore correction (body rotates slowly). Then:
// strutCompVel = mountVelStrut - hubVelStrut   (positive = mount approaching hub = compressing)
```

### contact_normal_split — D-06 force decomposition
```javascript
// [VERIFIED: codebase, physics.js queryContacts contract; math derived from first principles]
// Inside the hubContacts loop in stepSuspensionSubsteps:
const body_up = params._rotateVector({ x: 0, y: 1, z: 0 })
const bodyUpDot    = c.normal.x * body_up.x + c.normal.y * body_up.y + c.normal.z * body_up.z

  params.tireStiffness * c.depth + params.tireDamping * compressionVel
)
const Fn_strut = tireFnAtContact * bodyUpDot    // drives hub ODE (replaces Fn * c.normal.y)
const Fn_XZ_x  = tireFnAtContact * (c.normal.x - bodyUpDot * body_up.x)
const Fn_XZ_y  = tireFnAtContact * (c.normal.y - bodyUpDot * body_up.y)
const Fn_XZ_z  = tireFnAtContact * (c.normal.z - bodyUpDot * body_up.z)
// Accumulate XZ residual for body:

params._hubNormalXZ[i].z += Fn_XZ_z / N
```

### static_equilibrium_strut — D-11 init formula
```javascript
// [VERIFIED: derived; confirmed numerically: strutComp ≈ 111mm at current params]
// Returns strutComp[4] for vehicleState initialization
function computeStaticEquilibriumStrut(p) {
  const g = 9.81
  const strutComp = [0, 0, 0, 0]
  for (let i = 0; i < 4; i++) {
    const isFront  = i < 2
    const k_S      = isFront ? p.suspensionStiffnessFront : p.suspensionStiffnessRear
    const sprung   = p.mass * (isFront ? p.weightFront : p.weightRear) / 2
    strutComp[i]   = sprung * g / k_S   // D-11
  }
  return strutComp
}
// D-11 result at current params: strutComp ≈ 0.1112 m (111 mm) all corners
// L_S: front 0.20m, rear 0.22m — strutComp well within travel limits. OK.
```

## Open Questions (RESOLVED)

1. **Contact patch position for `_hubNormalXZ` torque arm in physics.js** (RESOLVED)
   - What we know: D-06a says torque arm = `contactPoint − bodyCG`. `contactPoint` is available inside `suspension.js` from `queryContacts` return.
   - What's unclear: `suspension.js` currently does not expose `contactPoint` to `physics.js`. The XZ residual is accumulated without its torque arm. Physics.js needs to either (a) receive the contact point per corner or (b) approximate it as the hub world position.
   - **Resolution:** approximate as the hub world position (option b). Plan 02 Task 3 derives `hub_i = getWheelPosition(i, vehicleState, params)` and uses `rHub_i = hub_i - vehicleState.position` as the torque arm. The ~0.37 m offset along the contact normal (hub to contact patch) contributes a small torque-arm error acceptable for body forces, documented in a code comment. The cleaner `_hubContactPoint[i]` accumulator is deferred to a future phase if torque-arm accuracy proves insufficient.

2. **`suspensionBodyOffsetFront/Rear` interaction with static equilibrium** (RESOLVED)
   - What we know: D-08 adds a Y offset for the mount in body space. The IC computation currently derives mount Y from `-(cgHeight - wheelRadius)` hardcoded.
   - What's unclear: If the body offset slider changes from its default of 0, the IC formula is no longer correct. The car will settle dynamically but may spawn with visible bounce.
   - **Resolution:** default value is `0.0` for both `suspensionBodyOffsetFront` and `suspensionBodyOffsetRear` per D-08, so the current spawn behavior is preserved. The slider shifts ride height but doesn't affect spawn correctness at IC because the IC scenarios use the default zero offset. Interactive slider changes after spawn will cause a small transient (dynamic settle) — acceptable per Claude's Discretion and not a regression-test failure mode.
