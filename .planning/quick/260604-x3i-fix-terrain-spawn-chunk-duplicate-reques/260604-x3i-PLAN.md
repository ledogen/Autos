<objective>
Fix the spawn-chunk duplicate-request race in `src/terrain.js` that orphans terrain meshes, leaving bumpy patches that the amplitude slider can never flatten.

Purpose: When the Terrain Amplitude slider is set to 0, spawn-chunk geometry stays frozen at the default amplitude (0.1) because orphaned, untracked meshes remain in the scene. `rebuildAllChunks()` only iterates `_chunkMap`, so it cannot touch them.

Output: A race-free, idempotent chunk lifecycle where every in-flight chunk key stays reserved until its geometry is actually built, and any double-build disposes the stale mesh instead of orphaning it.
</objective>

<root_cause>
Duplicate-request race in the chunk lifecycle:
1. `_updateChunkRing` requests a chunk only if `!_chunkMap.has(key) && !_pendingWorker.has(key)`, then adds the key to `_pendingWorker` (~line 352-355).
2. The worker `onmessage` handler IMMEDIATELY runs `_pendingWorker.delete(key)` and pushes to `_pendingQueue` (~line 213-214).
3. `_flushPendingQueue` builds only MAX_BUILDS_PER_FRAME=2 chunks/frame and only then runs `_chunkMap.set(key, …)` (~line 417).

Between the worker reply and the actual build, a key is in NEITHER `_pendingWorker` NOR `_chunkMap`. `_updateChunkRing` (every render frame) re-requests it → it is generated and built again → `_chunkMap.set` overwrites the reference, orphaning the FIRST mesh (added via `_scene.add` but no longer tracked). At spawn, 25 chunks queue at once and drain at 2/frame (~13 frames), so the window is wide and many chunks orphan. `rebuildAllChunks()` iterates only `_chunkMap`, so orphans stay frozen at amplitude 0.1.
</root_cause>

Two coordinated edits, both in src/terrain.js. Goal: a chunk key stays reserved in `_pendingWorker` for its whole in-flight lifetime, and the geometry build can never orphan a mesh.

EDIT A — keep the key reserved through the build:
- In the worker `onmessage` handler (~line 211-215), REMOVE the `this._pendingWorker.delete(key)` line. The handler should only push `{ key, cx, cz, heights }` onto `_pendingQueue`. Update the adjacent comment to note the key stays reserved until `_flushPendingQueue` builds it.
- In `_flushPendingQueue` (~line 385-418), AFTER `this._chunkMap.set(key, { mesh, heights })`, add `this._pendingWorker.delete(key)`. This is the single point where the reservation is released, and it happens only once the chunk is tracked in `_chunkMap`. This keeps the `!this._pendingWorker.has(key)` guard in `_updateChunkRing` effective for the entire request→built window, closing the race.

EDIT B — defensive idempotent build (guards against any future double-build):
- In `_flushPendingQueue`, BEFORE `this._scene.add(mesh)` / `this._chunkMap.set(...)`, check if `_chunkMap` already has an entry for `key`. If it does, remove the existing mesh from the scene and dispose its geometry first, mirroring the existing pattern in `_updateChunkRing`: `const stale = this._chunkMap.get(key); this._scene.remove(stale.mesh); stale.mesh.geometry.dispose()`. Do NOT dispose `this._material` (it is the shared MeshPhongMaterial). Add a brief comment explaining this makes the build idempotent so a stale orphan can never accumulate.

Preserve all existing conventions: heavy JSDoc, threat-mitigation comment style (e.g. T-06-xx), ES6 module syntax, no semicolons-as-needed style already in the file. Do NOT touch WORKER_SOURCE, the noise function, sampleHeight, sampleNormal, rebuildAllChunks, _updateChunkRing's request/dispose logic, or any physics path.
