## X/Z Normal Force Routing

### Q1: How should the X/Z contact normal force reach the body?

| Option | Description | Selected |
|--------|-------------|----------|
| Force + torque at hub world position | Apply F_XZ + r×F torque. Physically correct: side-load on tire creates yaw/roll moment. | ✓ |
| Force only at body CG | Simpler but incorrect: no rotation from side-loads. | |

### Q2: Where in the pipeline does the X/Z force+torque get accumulated?

| Option | Description | Selected |
|--------|-------------|----------|
| Inside stepSuspensionSubsteps, accumulated per substep | New `_hubNormalXZ[i]` accumulator. Consistent with `_tireFz[]` / `_suspForceAccum[]` pattern. | ✓ |
| Directly in physics.js after the substep call | Requires re-querying contacts. No new accumulators. | |

### Q3: Should X/Z force also update hub dynamics, or just go to body?

| Option | Description | Selected |
|--------|-------------|----------|
| Body only — hub has no lateral DOF | Hub stays 1D strut state. X/Z transmits through rigid constraint. | ✓ |
| Hub lateral too — add hubVx/hubVz | Scope creep, 3D hub already deferred. | |

**User's clarification:** "The hub should have its own mass (really tire + wheel mass), should be on a rigid slider in body y but still transfer x and z forces and associated torque based on position of contact patch and body cg." — confirms the rigid-slider interpretation.

### Q4: Spring reaction only, or contact normal Y also goes directly to body?

| Option | Description | Selected |
|--------|-------------|----------|
| Spring reaction only — contact normal Y drives hub, spring drives body | Avoids double-count per physics.js:270. | ✓ |
| Contact normal Y also goes directly to body | Would double-count on flat ground. | |

### Q5: Strut-axis projection: body_down · normal or always world-Y?

| Option | Description | Selected |
|--------|-------------|----------|
| Strut-axis projection: body_down · normal | Physically correct under tilt. | ✓ |

**User's clarification:** "The non vertical normal force y will have x and z components that should get transferred to the body. The reduced y will push less on the hub than a vertical but that vertical force will eventually be coupled to the hub through the suspension (integrated over time of course)."

### Q6: For X/Z torque on body, what position vector r?

| Option | Description | Selected |
|--------|-------------|----------|

## Droop Stop Travel Range (Suspension Parameter Shape)

### Q1: How much droop travel?

**User's freeform response:** "Variables and structures I want — main spring and damper: stiffness, damping and rest length. I want a slider to control the total travel of the suspension before bump stopping is applied. I want a body offset so I can locate that travel in space relative to the car body. For the bump stops I want stiffness and damping controls, they will always engage at 'full length' i.e. there is no precompression."

This redefined the parameter shape entirely:
- Main spring/damper (existing)
- `suspensionTravel` (new) — total travel before bump stop
- `suspensionBodyOffset` (new) — ride height control
- `bumpStopStiffness` (new) — slider
- No droop-stop slider (baked constant)

### Q2: Symmetric travel or separate bump/droop?

**User's freeform response:** "I'm pretty sure with rest length and stiffness we can already control the bump travel vs droop. Let's simplify a bit and just do a simple fixed spring force value. All it has to do is cancel the downward force of the suspension spring and the mass of the hub, and it can rely on the suspension damping to prevent oscillation. Set to a reasonable value for a hub that weighs twice our current one. We don't need to be able to adjust droop stop stiffness. Keep all sliders for bump stop stiffness."

Result: droop stop is a fixed `DROOP_STOP_STIFFNESS` constant sized for ~36 kg hub. No droop slider. Bump stop stiffness remains slider-exposed.

### Q3: What does 'body offset' control?

| Option | Description | Selected |
|--------|-------------|----------|
| Mount point height in body space (ride height) | Shifts mount Y relative to body CG. | ✓ |
| CG height offset (shifts body mass center) | More global; changes weight distribution. | |

### Q4: Single body offset or separate front/rear?

| Option | Description | Selected |
|--------|-------------|----------|
| Separate front/rear offsets | suspensionBodyOffsetFront/Rear. Independent rake tuning. | ✓ |
| Single offset | One value for all 4 corners. | |
