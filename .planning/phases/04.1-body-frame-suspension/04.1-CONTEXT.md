# Phase 4.1: Body-Frame Suspension - Context

**Gathered:** 2026-06-01
**Status:** Stub — awaiting full discuss-phase pass before planning

<domain>
## Phase Boundary

Phase 4.1 corrects the coordinate-frame error introduced in Phase 4: hub state is currently a scalar world-Y position (`hubY[i]`, `hubVy[i]`), and the suspension spring acts purely along world Y. This is physically wrong as soon as the body tilts — the strut should compress along the body's local up axis, not world Y. The most visible artifact is that ARB equilibrium (compression-matched in world Y) does not correspond to the physical equilibrium (twist-zero in body frame), allowing the inside wheel to be lifted past the outer wheel's body-frame height during cornering.

Phase 4.1 also adds bump and droop stops, since those are intrinsically defined in body-frame strut coordinates and would be throwaway work in the world-Y representation.

Phase 4.1 does NOT change the Pacejka tire model, does NOT change the ARB topology (still parallel-spring approximation per Phase 4 D-07), and does NOT introduce a 3D hub state — the hub remains constrained to a 1D coordinate along the strut axis.

</domain>

<motivating_artifacts>
## Why This Phase Exists

Three artifacts observed during Phase 4 review (2026-06-01) trace to the same root cause:

1. **Hub "floats" in flight (jump log `1780296563738.json`).** Fixed in-place by switching `src/suspension.js` from implicit Euler to N=4 explicit Euler. Bug was discretization-only; model was already correct. Not a Phase 4.1 item.

2. **ARB lifts inside hub past outer in body frame (rollover log `1780296509354.json`).** Root cause: `suspComp = L_S − (mountWorldY − hubY)` is the world-Y projection of strut length, not the strut length itself. ARB twist-zero is computed in this projection, so a rolled body produces an equilibrium that doesn't match the physical twist-zero. This phase's primary target.

3. **Suspension forces meaningless past ~45° roll.** Springs push along world Y, not along the strut axis. Once the body is on its side, the spring force is perpendicular to the strut and stops modeling anything. This phase's secondary target.

Plus: no bump stop and no droop stop. Currently the hub can travel arbitrarily far in either direction. Folded into this phase because the stop positions are body-frame quantities.

</motivating_artifacts>

<decisions>
## Implementation Decisions (DRAFT — to be locked in discuss-phase)

### Coordinate frame
- **D-01 (proposed):** Replace `hubY[i]`, `hubVy[i]` state with `strutComp[i]` (m, positive = compressed below rest length) and `strutCompVel[i]` (m/s, positive = compressing). 1D state per corner along the body-local strut axis.
- **D-02 (proposed):** Hub world position is derived, not state: `hubPos = mountWorldPos + (L_S − strutComp[i]) · body_down_world`, where `body_down_world = (0,−1,0)` rotated by the body quaternion.
- **D-03 (proposed):** Mount velocity along strut axis: project `(bodyVelocity + ω × rMount)` onto `body_down_world` to get `mountVelStrut`. Compression velocity `strutCompVel = −(hubVelAlongStrut − mountVelStrut)`. Sign chosen so positive = compressing.

### Forces
- **D-04 (proposed):** Spring, damper, and ARB all act along the strut axis (body-local up/down). Magnitudes computed identically to Phase 4 (`k·strutComp`, `c·strutCompVel`, `k_arb·Δstrut`); only the direction of application changes.
- **D-05 (proposed):** Gravity on hub along strut axis: `m_u · g · (world_up · body_up)` projected component. When body is upright, this is full `m_u·g` pulling hub away from body. When body is horizontal (on its side), this tapers to zero. Lateral component of gravity acting on hub is absorbed into the strut constraint (not modeled as an inertial term; small-mass approximation).
- **D-06 (proposed):** Tire contact normal force projected onto strut axis for the hub ODE. Pacejka feed (`_tireFz[i]`) unchanged — still the magnitude of the tire spring force at the contact patch.

### Bump and droop stops
- **D-07 (proposed):** Add `suspensionBumpStop{Front,Rear}` (m, compression at which bump stop engages — typical 60–80% of `suspensionRestLength`) and `suspensionDroopStop{Front,Rear}` (m, extension limit beyond rest length).
- **D-08 (proposed):** Stop force model: linear penalty spring with stiffness `bumpStopStiffness` (≈10× main spring) once `strutComp` exceeds `bumpStop` or falls below `−droopStop`. Stiff enough to look hard, soft enough to integrate stably at the N=4 substep rate.

### Integrator
- **D-09 (proposed):** Inherit Phase 4's N=4 explicit Euler substep choice. Re-verify stability with the new bump-stop stiffness; raise N if needed.

### State initialization
- **D-10 (proposed):** Static equilibrium: `strutComp[i] = (m·g·weight_i/4) / k_S_i` at rest. Computed at IC load / scenario reset, same hook as Phase 4 hubY initialization.

### Migration / compatibility
- **D-11 (proposed):** Existing logged fields (`fl_fz`, etc.) retain their meaning (tire spring Fz). Add `strutComp` per-corner to wheelDebug + logger for diagnostic visibility.
- **D-12 (proposed):** Scenarios with hard-coded `hubY` initial conditions need a migration pass. Audit `scenarios/*.json` for any reference; rewrite as `strutComp` if found.

### Claude's Discretion
- Whether to fold bump/droop stops into the same phase or split (current draft: same phase, since they're co-located code changes)
- Bump-stop spring stiffness starting value
- Exposure of bump/droop stop positions as sliders vs fixed params
- Whether to expose `wheelMass` projected mass terms or assume small-mass approximation throughout
- Naming: `strutComp` vs `suspComp` (current Phase 4 uses `suspComp` as a derived value; reusing the name as state could mislead future LLM sessions)

</decisions>
