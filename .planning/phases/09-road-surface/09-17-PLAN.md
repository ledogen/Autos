<objective>
Kill the bounce on visually-smooth road. The car bounces because the PHYSICS road
height is a staircase while the VISUAL ribbon is continuous: `queryNearest`'s
`probeSpline` returns the NEAREST DISCRETE SAMPLE (u = i/n, ~2 m spacing) and then
evaluates `point = getPointAt(bestU)` at that discrete u. As the tire-contact point
moves, `nr.point.y` snaps between discrete samples → `_sampleCarveWorld` reads a
piecewise-constant designY → the suspension is kicked by invisible ~2 m steps. This
is a 09-13 regression (physics moved off the interpolated smoothed grade onto the
discrete `nr.point.y`).

Purpose: Make the physics road surface C0-continuous and match the continuous visual
ribbon (road-mesh.js sweepRibbon, which samples `spline.getPointAt(u)` finely).
Output: Refined `queryNearest` (projection to a continuous u), an extended
continuity harness that gates the fix, and a human-verified smooth ride.

OUT OF SCOPE (do NOT touch): camber rate-limiting (09-14 owns it), dirt road
(09-15), Phase 8 routing (canonical run, `_limitCurvature`, slicing), and the
visual ribbon's sampling in road-mesh.js.
</objective>

    In queryNearest, after probeSpline has settled bestSpline / bestU / bestArcLen on the
    sliced-spline path (the `if (bestSpline)` block, ~461), REFINE bestU to a continuous
    parameter by projecting the query point (wx,wz) onto the local polyline near bestU —
    do NOT increase the probe sample count.

    Algorithm (operate in XZ only; keep allocations off the hot path by reusing _scratchPt
    plus at most a couple of cheap locals — no new THREE.Vector3 per query beyond the two
    already returned):
    1. Derive the per-sample step du = 1/n. probeSpline currently scopes n locally; lift n
       (or recompute du from bestArcLen the same way: n = max(16, min(256, ceil((bestArcLen||64)/2))),
       du = 1/n) so the refine knows the spacing that produced bestU.
    2. Bracket bestU with its neighbours: uPrev = clamp(bestU - du, 0, 1),
       uNext = clamp(bestU + du, 0, 1). Evaluate the three bracket points with
       getPointAt into scratch locals (read .x/.z each). bestU itself is the middle.
    3. Project (wx,wz) onto each of the two XZ segments [prev→mid] and [mid→next]:
       for segment from A to B, t = clamp(((wx-A.x)*(B.x-A.x) + (wz-A.z)*(B.z-A.z)) /
       max((B.x-A.x)^2 + (B.z-A.z)^2, 1e-12), 0, 1); projected XZ distance² to (wx,wz)
       at that t gives the segment's best fit. Pick the segment with the smaller distance².
    4. Map the winning (segment, t) back to a continuous refinedU: for the prev→mid segment
       refinedU = uPrev + t*(bestU - uPrev); for the mid→next segment
       refinedU = bestU + t*(uNext - bestU). Clamp refinedU to [0,1].
    5. Evaluate the returned values at refinedU: point = bestSpline.getPointAt(refinedU),
       tangent = bestSpline.getTangentAt(refinedU) (still UNIT). Set arcS = refinedU * bestArcLen
       so downstream (_sampleCarveWorld camber probe, SURF-06 pothole keying) stays consistent.
    6. Preserve the EXACT return contract: { point, tangent, runKey: bestRunKey, arcS, spline: bestSpline }.

    Leave the raw-network fallback (~471-501) UNCHANGED — it already interpolates between
    polyline points; the refine targets only the primary sliced-spline path. resolveSpawn and
    other callers consume the same contract and benefit from the more accurate projection.
    Update the queryNearest JSDoc (~399-416) to state that the sliced-spline point is now a
    continuously-projected point (C0), not the nearest discrete sample. Do NOT change
    _sampleCarveWorld, crown/camber/pothole math, or sweepRibbon.

    Worker sync constraint: this edit is in road.js only — terrain-worker.js MUST stay
    byte-identical. Do not edit it.
